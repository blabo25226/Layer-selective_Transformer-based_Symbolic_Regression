# AGENTS.md — LTSR研究リポジトリのAI作業指針

このファイルは、このリポジトリで作業する **Cursor、Claude Code、Claude Science、Codex、およびその他のAI** に対する共通説明書である。
目的は、複数のAIが交代・並行して作業しても、研究設計、コード、結果、Git履歴の整合性を保つことにある。

このルールはリポジトリ全体に適用する。下位フォルダに別の `AGENTS.md` がある場合は、そのフォルダ内では下位ファイルの指示を追加で守る。
ユーザーから明示的な指示がある場合は、ユーザーの指示を優先する。

## 1. この研究を一文で説明すると

事前学習済みTransformer型シンボリック回帰モデル **NeSymReS** を遺伝子制御方程式へ適応させ、
寄与の大きい少数のTransformer層だけをfine-tuningする方法を、全層学習、ランダム層学習、PySR、TPSRと比較する研究である。

中心となる推定問題は、遺伝子 (i) の発現量を (x_i(t)) として、

$$
\frac{dx_i}{dt}=f_i(x_1,\ldots,x_p)
$$

の未知関数 (f_i) を、人間が読める数式として発見することである。

## 2. 作業開始時に必ず確認するもの

タスクに着手する前に、最低限次を確認する。

1. `git status --short` と `git branch --show-current`
2. [`README.md`](README.md)：研究背景、現状、CPU予備結果、重要な限界
3. [`plan/20260714_firstplan.md`](plan/20260714_firstplan.md)：研究質問と当初計画
4. GPU実験に関係する場合は [`GPU_RUN.md`](GPU_RUN.md)
5. 図表を作る場合は [`graphs/README.md`](graphs/README.md)
6. 対象Phaseの `results/phase_results/*_report.md` と実行スクリプト
7. 変更対象に対応する `tests/` のテスト

過去の会話やAIの要約だけを根拠にせず、必ず現在のファイルを読むこと。ブランチは作業ごとに確認し、`main` が常に最新だと仮定しない。

## 3. 現在の研究状態

- CPU上でPhase 0–8のパイプラインと小規模な予備実験を実施済み。
- GPUによる多seed・大規模な本実験は未実施。
- READMEにあるCPU結果の一部は、最新の評価設計修正より前に生成された **legacy pilot** である。
- 最新コードでは、validation/test分離、paired seed、Studentのt区間、trajectory分割、failure-aware集計などを修正済み。
- CPU pilotの数値は研究の方向確認には使えるが、論文レベルの仮説確証として扱わない。
- `gpu-scale-prep` はGPU本実験の準備と再現性強化を含む作業ブランチである。`main` との関係は毎回Gitで確認する。

研究状態を更新した場合は、コードだけでなくREADME、GPU_RUN、該当Phaseレポートのどれを更新すべきか検討する。

## 4. 研究上の絶対条件

以下は、速度や見栄えより優先する。

### 4.1 データリークを起こさない

- 層ランキング、hyperparameter、early stopping、モデル選択には **validation** を使う。
- **test** は方法をすべて固定した後の最終評価だけに使う。
- Phase 4でtestを層選択に使わない。
- Phase 5以降で、test結果を見てtop層や設定を選び直さない。
- 合成データは、可能な限り行単位ではなくmotif・式構造・問題単位で分割する。

### 4.2 時系列の分割順を守る

DREAM4などの時系列は、trajectoryをtrain/validation/testへ分けてから有限差分を計算する。
有限差分を作った後に行をランダム分割すると、隣接時点の情報が別splitへ漏れる可能性がある。

### 4.3 条件比較を公平にする

- 同じseed内のtop/random/bottom/full条件では、乱数状態、データ順、初期checkpointをそろえる。
- hyperparameter探索では各trainable条件へ同じ候補数を与え、validationだけで選択する。選択後のtest評価は一度だけ行う。
- 条件ごとに異なるデータ順を引いて差を水増ししない。
- seed集合、学習回数、decode budgetを記録する。
- PySR、beam、TPSRの比較では、計算時間または候補評価回数の差も報告する。

### 4.4 生成失敗を隠さない

- decode失敗、構文エラー、NaN、Inf、評価不能な式を中央値から単純除外しない。
- `valid_rate` とfailure-penalizedな指標を主結果に含める。
- 成功した式だけの条件付き指標は `valid_nmse`、`valid_r2` など、条件付きだと分かる名前にする。

### 4.5 数値精度と式回復を区別する

低NMSEは、真の数式構造や生物学的機構を回復したことを意味しない。最低限、次を分けて評価する。

- ID/OOD NMSE、(R^2)
- exact / skeleton / symbolic equivalence
- variable precision、recall、F1
- 式複雑度
- valid rate
- 特異点、外挿安定性、非負性などの安全性

### 4.6 統計を過大解釈しない

- 少数seed・少数donorの95%信頼区間にはStudentのt分布を使う。
- (n=2)、単一seed、単一network、単一holdoutの結果を一般的結論にしない。
- 平均・中央値・標準偏差・信頼区間のどれを報告しているか明記する。
- 有意差がない結果、仮説を支持しない結果、逆転した結果も残す。

### 4.7 実データへの主張を限定する

- DREAM4の有限差分ターゲットは真のODE微分ではない。
- GSE112372は5時点・4 donorsの小規模なapplication demoである。
- ヒトデータの式を「真の制御ODE」や「因果機構」と断定しない。
- `tan`、危険な除算、特異点を含む式は、NMSEが良くても生物学的に妥当とは扱わない。

## 5. 実装方針

### 5.1 既存構造を再利用する

- 共通処理は `src/`、実験入口は `scripts/`、設定は `configs/`、テストは `tests/` に置く。
- Phaseスクリプト間でsplit、seed、集計、メトリクスを複製せず、既存の共通モジュールを使う。
- 既にある `src/data/splits.py`、`src/evaluation/aggregation.py`、`src/experiment_runtime.py` などを先に確認する。
- NeSymReSとTPSRの参照実装は `NSRS/` と `TPSR/` にある。必要性のない大規模改変を避ける。
- 参照実装へ互換修正を入れる場合は、理由と上流との差をコメントまたはコミットに残す。

### 5.2 再現性を保つ

- Python、NumPy、PyTorch、DataLoaderで使用する乱数を明示的にseedする。
- runごとにGit branch/commit、環境、checkpoint SHA256、引数、主要設定をmanifestへ保存する。
- checkpoint名だけを信用せず、アーキテクチャとstate dictを確認する。
- 結果を手作業でJSONへ転記しない。可能な限り実行コードから直接保存する。
- 集計値だけでなく、問題単位の予測と失敗理由を保存する。
- 層寄与度では正規化前のbase/full/各層スコアもseedごとに保存する。full FTがbaseを改善しない指標を、符号反転した正規化寄与度として順位へ混ぜない。

### 5.3 出力数式を必ず残す

シンボリック回帰を実行するコードは、各問題について最低限次を保存する。

- `eq_id`
- 真の式（存在する場合）
- モデルが出力した生の式
- 簡約後の式
- 変数名と実遺伝子名の対応
- NMSE、(R^2)、variable F1、complexity、valid判定
- 失敗した場合の理由

集約値だけを保存して式を捨てない。READMEやレポートには、成功例だけでなく代表的な失敗式も載せる。

### 5.4 コード品質

- Python 3.10を基準にする。Hydra互換性のため、Python 3.12を本実験環境へ勝手に変更しない。
- 型、docstring、変数名は、研究上の意味が分かるように書く。
- 暗黙のfallbackで実験条件を変えない。checkpoint、CUDA、入力データ、設定不足は明確にfail fastする。
- 例外を握りつぶさない。継続可能な問題単位の失敗は記録し、run全体の致命的エラーは停止する。
- 新しい挙動には対応するテストを追加する。
- テストのための生成物は一時ディレクトリへ置き、リポジトリへ残さない。

## 6. ファイルと成果物の扱い

### 6.1 データ

- `data/` は原則Git管理外である。
- データ本体、checkpoint、個人情報、巨大な生成物をコミットしない。
- 公開データでも、取得元、accession、前処理、checksumまたは版を記録する。
- `data/` という名前のignore規則が `src/data/` を誤って除外しないか注意する。

### 6.2 結果

- 既存のCPU pilotを上書きしない。
- GPU runは原則 `results/runs/<run-id>/` に保存する。
- runの途中結果と完了結果を区別し、manifestのstatusを更新する。
- レポートの数値は保存済みJSON/CSVから生成または照合する。
- 古い評価設計の結果と最新コードの結果を同じ表で混ぜる場合は、世代を明記する。

### 6.3 図と表

独立した図表は必ず次へ置く。

```text
graphs/<run-id>/figures/
graphs/<run-id>/tables/
```

CPU pilotの可視化は `graphs/cpu_pilot/` を使う。ファイル名にはPhaseと内容を含める。
外部ライブラリや参照実装に元から含まれる画像を、研究成果の `graphs/` へ移動しない。

### 6.4 文書

- READMEは高校生でも大筋を追える日本語を保つ。
- 専門語は初出で説明し、略語だけで進めない。
- 数式が必要な説明にはTeXを使う。
- 文献は原論文、公式ドキュメント、公式データページを優先し、URLと書誌情報を確認する。
- AIが生成した架空の論文、DOI、数値、引用を残さない。
- CPU pilotとGPU本結果、観測事実と推測、支持された仮説と未支持の仮説を区別する。

### 6.5 GitHub MarkdownとTeXの注意点

READMEはローカルエディタのプレビューだけでなく、GitHub上で正しく表示される必要がある。
このリポジトリでは、MarkdownとTeXが衝突しやすい箇所について次の形式を使う。

#### インライン数式

添字 `_`、波括弧 `{}`、アスタリスク `*`、バックスラッシュを含む式は、単純な `$...$` ではなく、
GitHubのbacktick付き数式区切りを使う。

```text
Good: $`D=\{(\mathbf{x}_i,y_i)\}_{i=1}^{N}`$
Avoid: $D=\{(\mathbf{x}_i,y_i)\}_{i=1}^{N}$
```

単純な式でも、README内では形式を混在させず、原則として `$` とbacktickを組み合わせた形式へ統一する。
TeXソースの `_` をMarkdown用に `\_` へ変えない。数式区切りを正しく使い、TeXとして `_` を記述する。

#### 表示数式

複数行または独立表示する式には、`$$` ではなくGitHubの `math` コードフェンスを使う。

````text
```math
f(x)=x^2
```
````

通常のコードブロックへTeXを書いても数式にはならない。開始フェンスは必ず `math` を指定する。

#### TeXコマンド

このREADMEのGitHub表示では、バックスラッシュ付きの `operatorname` コマンドがレンダリングエラーの原因になったため、
READMEでは一切使用しない。演算子名やローマン体のラベルには `\mathrm{...}` を使う。
新規追加だけでなく、編集後のREADME全体で禁止コマンドの残存が0件であることを確認する。

```text
Good: \mathrm{NMSE}, \mathrm{softmax}, \mathrm{arg\,min}
Avoid: operatorname系コマンドによる演算子名の記述
```

新しいTeXコマンドを追加するときは、GitHubのMathJaxで実際にレンダリングできるか確認する。

#### 太字と括弧

`**...**` の内側に半角括弧 `()` または全角括弧 `（）` がある場合、太字記号の外側を周囲の文章へ密着させない。
開始側と終了側に通常の文章が続く場合は、半角空白を入れる。

```text
Good: これは **シンボリック回帰（SR）** の説明である。
Avoid: これは**シンボリック回帰（SR）**の説明である。
```

行頭・行末では存在しない側の空白は不要だが、直後に句読点やコロンがある場合もGitHub表示を確認する。

#### 文書変更後の確認

最低限、次を検索する。

```bash
rg -n '\\' README.md | rg 'operatorname|operatename'
rg -n '^\$\$$' README.md
rg -n '\*\*[^*]*(\(|（)[^*]*(\)|）)[^*]*\*\*' README.md
```

1つ目の禁止コマンド検索は必ず0件にする。2つ目は表示数式の区切りを確認し、
3つ目は太字の外側に必要な空白があるか目視確認する。
数式を大きく変更した場合は、GitHub上のREADMEまたはGitHub Markdown APIで、inline/display mathとして認識されることを確認する。

## 7. 検証手順

変更の大きさに応じて、次を実行する。

### 最低限の静的確認

```bash
python -m compileall -q src scripts tests
```

### 標準テスト

```bash
python -m pytest -q
```

### GPU実験前

```bash
python scripts/preflight_gpu.py --checkpoint /path/to/model.ckpt
bash -n scripts/run_gpu_pipeline.sh
```

GPUを持たないPCではCUDA実験を成功したように装わない。CPUで確認できた範囲と、GPUで未確認の範囲を明示する。
長時間の本実験を開始する前にsmall/smoke設定で、出力先、manifest、式保存、resume/failure動作を確認する。

## 8. Git運用

- 作業前後に `git status --short` を確認する。
- ユーザーまたは他AIの未コミット変更を、自分の変更だと仮定しない。
- 関係のない差分を修正、整形、削除、stageしない。
- `git reset --hard`、無断のcheckoutによる破棄、履歴改変を行わない。
- ファイル移動はGit履歴への影響が大きいため、機能変更と分けて行う。
- 実装・修正は意味のまとまりごとに小さくcommitする。
- commit messageは何を変更したか分かる英語の命令形または簡潔な要約にする。
- push、merge、rebase、PR作成は、ユーザーの依頼または明確な運用文脈がある場合に行う。
- commit前に `git diff --check` と必要なテストを実行する。

複数AIが同じworktreeを使う場合、他AIの作業中ファイルを上書きしない。作業範囲が重なる場合は、先にユーザーへ報告するか、
変更を小さく区切り、現在の差分を再読してからpatchを当てる。

## 9. AIごとの推奨役割

これは固定的な担当分けではない。どのAIも必要な作業を担当できるが、引き継ぎ時の目安とする。

### Cursor

- IDE上での局所的な実装、補完、参照検索、短い反復に向く。
- 生成コードをそのまま採用せず、既存共通関数、split、出力形式、テストとの整合性を確認する。
- 複数ファイルへ広がる変更では、変更一覧と未検証点を残す。

### Claude Code

- リポジトリ横断レビュー、バグ調査、テスト追加、別実装の検討に向く。
- レビューでは「研究設計上の問題」と「単純な実装バグ」を分ける。
- 修正を加えた場合、レビュー指摘のどれを直し、どれを保留したかを明記する。

### Claude Science

- 文献調査、仮説の妥当性、実験計画、統計、科学的主張の批判的検討に向く。
- 提案をコードへ反映する前に、利用可能なデータ、モデル制約、計算資源、既存評価設計と照合する。
- 文献・DOI・引用・最新研究は原典で検証し、推測を事実として記載しない。
- 科学的に望ましい案と、このリポジトリですぐ実装可能な案を分けて提示する。

### Codex

- リポジトリ全体の統合、実装修正、テスト、再現性、ファイル整理、Git commitまでの一連の作業に向く。
- 既存差分を保護し、変更後はコードだけでなくレポート・実行手順・成果物規約との整合性を確認する。
- レビュー時は実ファイルと実行結果を根拠にし、未実行のGPU計算を完了扱いにしない。

## 10. AI間の引き継ぎ形式

作業終了時は、次の情報を短く残す。チャット、commit、レポートのいずれでもよいが、後続AIが確認できる形にする。

```text
目的:
作業ブランチ / commit:
変更したファイル:
実装・判断した内容:
実行したテストと結果:
生成した成果物:
未実行・未確認:
既知の問題 / 次の作業:
```

レビューだけを行った場合は、指摘を重大度順に並べ、各指摘にファイル・行・根拠・研究結果への影響を付ける。
実装した場合は、成功したことだけでなく、試して失敗した方法や残った制約も共有する。

## 11. 完了条件

「コードが動いた」だけでは完了ではない。タスクの種類に応じ、次を満たす。

### コード変更

- 要求された挙動を実装した。
- 対応テストを追加または実行した。
- 研究上の絶対条件を破っていない。
- 出力形式、README、GPU_RUN、設定の更新要否を確認した。
- 差分に無関係な変更がない。

### 実験実行

- run設定、seed、commit、環境、checkpointを記録した。
- 集約値と問題単位結果、出力式、失敗を保存した。
- validationで選択し、testは最終評価だけに使った。
- 結果を過大解釈せず、制約と未支持仮説を報告した。

### 文書・レビュー

- 現在のコードと保存済み結果に基づいている。
- 事実、推測、提案を区別している。
- 専門用語、数式、文献リンクが読者に追える。
- 実行していない検証を「確認済み」と書いていない。

## 12. 判断に迷ったときの優先順位

1. 研究結果の正しさとデータリーク防止
2. 保存済みデータ・ユーザー変更・Git履歴の保護
3. 再現性と失敗の可視化
4. 最小で検証可能な実装
5. 計算効率と作業速度
6. 見た目やコードの美しさ

不明点を安全にファイルやテストから確認できる場合は確認して進める。研究結論、データ利用、長時間GPU計算、
破壊的Git操作など、判断によって結果やコストが大きく変わる場合は、推測で進めずユーザーへ確認する。
